# 4장. ORM(Object Relational Mapping)

Query Builder는 SQL을 더 안전하고 구조적으로 만들었지만,  
여전히 데이터 접근의 중심은 SQL이었다.

개발자는 여전히

* 어떤 테이블을 조회할지
* 어떤 JOIN을 할지
* 어떤 조건을 사용할지

를 SQL 기준으로 생각해야 했다.

---

## 관점의 변화

여기서 하나의 질문이 등장한다.

👉 “데이터를 굳이 SQL로만 다뤄야 할까?”

애플리케이션은 객체 기반으로 동작한다.

```ts
const user = {
  id: 1,
  name: "test"
};
```

하지만 데이터베이스는 테이블 기반이다.

이 두 구조의 차이를 줄이기 위해 등장한 것이  
👉 **ORM(Object Relational Mapping)** 이다.

---

## 객체로 데이터를 다루는 방식

ORM에서는 데이터를 SQL이 아닌  
객체를 통해 접근한다.

대표적으로 Prisma 같은 도구를 보면  
이런 형태로 코드를 작성하게 된다.

```ts
const user = await prisma.user.findUnique({
  where: { id: userId }
});
```

이제 개발자는

* SELECT 문을 작성하지 않고
* 테이블을 직접 다루지 않는다

👉 대신 객체와 메서드를 통해 데이터를 다룬다

---

## 무엇이 달라졌는가

Query Builder와의 가장 큰 차이는  
👉 “데이터를 바라보는 기준”이다

* Query Builder → SQL 중심
* ORM → 객체 중심

즉,

👉 데이터 접근의 기준이 SQL에서 객체로 이동한다

---

## ORM의 핵심 특징

ORM은 다음과 같은 특징을 가진다.

* 객체 ↔ 테이블 매핑
* SQL 추상화
* 도메인 중심 코드 작성 가능

개발자는 더 이상 SQL을 직접 작성하지 않고  
객체를 다루듯 데이터를 처리한다.

---

## 코드의 변화

기존 방식과 비교하면 차이가 명확하다.

### Raw Query

```ts
await db.query("SELECT * FROM users WHERE id = $1", [userId]);
```

### Query Builder

```ts
await knex('users').where({ id: userId }).first();
```

### ORM

```ts
await prisma.user.findUnique({
  where: { id: userId }
});
```

👉 점점 SQL이 보이지 않게 된다

---

## ORM의 장점

이 방식은 생산성과 구조 측면에서 큰 변화를 만든다.

* 반복 코드 감소
* 비즈니스 로직과 데이터 접근 분리
* 코드 일관성 향상
* 객체 중심 설계 가능

특히 서비스 규모가 커질수록  
구조적인 이점이 크게 드러난다.

---

## 하지만 완전한 해결은 아니다

ORM은 강력하지만, 새로운 문제도 함께 가져온다.

* 실제 어떤 SQL이 실행되는지 파악하기 어려움
* 성능 제어가 제한됨
* 복잡한 쿼리에서 표현 한계

대표적인 문제로는

* N+1 문제
* Lazy Loading / Eager Loading 이슈

등이 있다.

---

## 추상화의 대가

ORM은 SQL을 숨겨준다.

하지만 그만큼

👉 내부 동작을 이해하지 못하면  
👉 예상하지 못한 성능 문제가 발생할 수 있다

즉,

* 편해진 만큼
* 보이지 않는 부분이 늘어난다

---

## 다음 단계로의 현실적인 선택

지금까지의 흐름을 보면

* Raw Query → 직접 제어
* Query Builder → 구조적 개선
* ORM → 추상화

로 발전해왔다.

하지만 하나의 방식이 모든 문제를 해결하지는 못한다.
