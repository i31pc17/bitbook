# 16장. AI 에이전트의 기본 개념

## 1. 챗봇과 에이전트는 다르다

AI를 처음 사용할 때는 보통 챗봇 형태로 시작한다.

사용자가 질문을 입력하면 AI가 답변한다.

```text
사용자:
Redis timeout이 자주 발생하는데 원인 후보를 알려줘.

AI:
Redis CPU 사용률, 메모리 사용량, slowlog, 네트워크 지연, 커넥션 수를 확인해보세요.
```

이 구조에서는 AI가 주로 설명하고 제안한다.

하지만 실제 업무에서는 단순히 답변만 받고 싶은 것이 아닐 때가 많다.

예를 들어 이런 요청을 생각해보자.

```text
최근 1시간 동안 Redis timeout이 늘어난 원인을 확인해줘.
```

이 요청을 처리하려면 AI가 단순히 일반적인 원인 후보를 말하는 것으로는 부족하다.

실제로 해야 할 일이 있다.

```text
- 모니터링 지표를 조회한다.
- Redis slowlog를 확인한다.
- 최근 배포 이력을 확인한다.
- 관련 에러 로그를 검색한다.
- 원인 후보를 정리한다.
- 추가 확인이 필요한 항목을 제안한다.
```

챗봇은 보통 “무엇을 확인해야 하는지” 알려준다.

에이전트는 한 단계 더 나아가, **도구를 사용해서 직접 확인하고 작업을 수행하는 AI 구조**에 가깝다.

```text
챗봇:
무엇을 해야 하는지 답변한다.

에이전트:
목표를 이해하고, 필요한 도구를 사용해, 여러 단계를 거쳐 결과를 만든다.
```

예를 들어 개발 업무에서는 에이전트가 다음처럼 동작할 수 있다.

```text
사용자:
이번 PR 변경 내용을 보고 리뷰 포인트 정리해줘.

에이전트:
1. GitHub에서 PR diff 조회
2. 변경 파일 분석
3. 테스트 파일 존재 여부 확인
4. 위험한 변경 탐지
5. 리뷰 코멘트 초안 작성
```

이처럼 에이전트는 단순 답변 생성보다 더 넓은 개념이다.

AI가 생각만 하는 것이 아니라, 필요한 도구를 호출하고 작업 흐름을 이어간다.

> 에이전트는 목표를 달성하기 위해 AI가 여러 단계를 계획하고, 도구를 사용하며, 결과를 만들어내는 구조를 말한다.  
> 단순 챗봇보다 업무 자동화에 더 가깝다.

## 2. 에이전트가 필요한 이유

일반적인 AI 챗봇은 많은 일을 도와준다.

하지만 챗봇은 사용자가 정보를 직접 제공해야 하는 경우가 많다.

예를 들어 로그 분석을 하려면 사용자가 로그를 복사해서 붙여 넣어야 한다.

```text
사용자:
아래 로그 보고 원인 분석해줘.

[로그 붙여넣기]
```

코드 리뷰를 하려면 변경 코드를 붙여 넣어야 한다.

```text
사용자:
아래 diff 보고 리뷰해줘.

[diff 붙여넣기]
```

문서 검색을 하려면 관련 문서를 검색해서 AI에게 전달해야 한다.

```text
사용자:
이 문서를 기준으로 답해줘.

[문서 붙여넣기]
```

이 방식은 유용하지만 한계가 있다.

```text
- 사용자가 필요한 자료를 직접 찾아야 한다.
- 여러 시스템을 오가며 정보를 모아야 한다.
- 반복 업무를 자동화하기 어렵다.
- AI가 실제 작업을 수행하지는 못한다.
```

에이전트는 이런 한계를 줄이기 위해 필요하다.

에이전트는 AI가 외부 도구와 연결되어 필요한 정보를 직접 조회하거나, 정해진 작업을 수행할 수 있게 한다.

예를 들어 장애 대응 에이전트를 생각해보자.

```text
사용자:
방송방 입장 실패가 늘어난 것 같은데 확인해줘.

에이전트:
- 모니터링 시스템에서 입장 API 에러율 조회
- 로그 시스템에서 timeout 로그 검색
- 최근 배포 이력 확인
- Redis 상태 확인
- 원인 후보와 조치안을 정리
```

개발 에이전트도 가능하다.

```text
사용자:
이 이슈에 맞는 PR이 있는지 찾아줘.

에이전트:
- Jira 이슈 조회
- GitHub PR 검색
- 커밋 메시지에서 이슈 키 검색
- 관련 PR 목록 정리
```

문서 관리 에이전트도 생각할 수 있다.

```text
사용자:
하트 충전 미반영 처리 문서가 최신인지 확인해줘.

에이전트:
- 문서 저장소에서 관련 문서 검색
- 마지막 수정일 확인
- 최근 장애 보고서와 내용 비교
- 오래된 표현이나 누락된 절차 표시
```

에이전트가 필요한 이유는 단순하다.

AI가 답변만 하는 수준을 넘어, 실제 업무 시스템과 연결되어 반복적인 확인 작업을 줄일 수 있기 때문이다.

## 3. 에이전트의 기본 구성 요소

에이전트는 보통 다음 요소로 구성된다.

```text
목표
→ 계획
→ 도구
→ 실행
→ 관찰
→ 결과 정리
```

하나씩 살펴보자.

### 3.1 목표

목표는 사용자가 원하는 최종 결과다.

```text
최근 장애 원인 후보를 정리해줘.
이번 PR 리뷰 포인트를 찾아줘.
고객 문의를 확인하고 답변 초안을 만들어줘.
```

목표가 모호하면 에이전트도 잘못 움직일 수 있다.

```text
나쁜 목표:
이거 처리해줘.

좋은 목표:
최근 1시간 동안 로그인 실패율이 증가한 원인 후보를 로그와 배포 이력을 기준으로 정리해줘.
```

### 3.2 계획

계획은 목표를 달성하기 위해 어떤 단계를 거칠지 정하는 것이다.

예를 들어 로그인 실패율 증가 원인을 확인하려면 다음 계획이 필요할 수 있다.

```text
1. 로그인 API 에러율 확인
2. 실패 원인 코드별 비율 확인
3. 최근 배포 이력 확인
4. DB 또는 Redis 장애 여부 확인
5. 원인 후보 정리
```

에이전트는 복잡한 작업을 한 번에 처리하기보다 단계별로 진행한다.

### 3.3 도구

도구는 에이전트가 사용할 수 있는 외부 기능이다.

```text
- 로그 검색 도구
- 모니터링 조회 도구
- GitHub 조회 도구
- Jira 조회 도구
- 문서 검색 도구
- DB 조회 도구
- 메일 발송 도구
- 캘린더 조회 도구
```

도구가 있어야 에이전트가 실제 정보를 확인할 수 있다.

도구가 없으면 AI는 일반적인 답변만 할 수 있다.

### 3.4 실행

실행은 에이전트가 도구를 호출하는 단계다.

```text
로그 검색 도구 호출
→ 최근 1시간 timeout 로그 조회

GitHub 도구 호출
→ 특정 PR diff 조회

문서 검색 도구 호출
→ 환불 정책 문서 검색
```

### 3.5 관찰

관찰은 도구 실행 결과를 읽고 다음 행동을 결정하는 단계다.

예를 들어 로그를 조회했더니 특정 에러가 많았다면, 그 에러와 관련된 문서를 추가로 검색할 수 있다.

```text
로그 검색 결과:
ERR_PAYMENT_409 증가

다음 행동:
ERR_PAYMENT_409 처리 문서 검색
최근 결제 API 배포 이력 확인
```

### 3.6 결과 정리

마지막으로 에이전트는 실행 결과를 사용자가 이해하기 쉬운 형태로 정리한다.

```text
## 확인된 사실
## 원인 후보
## 추가 확인 필요
## 권장 조치
## 참고한 로그/문서
```

좋은 에이전트는 단순히 도구를 많이 호출하는 것이 아니다.

목표에 맞게 필요한 도구를 사용하고, 결과를 근거와 함께 정리해야 한다.

## 4. 에이전트의 동작 흐름

에이전트는 보통 다음 흐름으로 동작한다.

```text
사용자 요청
→ 목표 이해
→ 필요한 작업 계획
→ 도구 선택
→ 도구 실행
→ 결과 확인
→ 추가 도구 실행 여부 판단
→ 최종 답변
```

예를 들어 사용자가 이렇게 요청했다고 하자.

```text
최근 배포 이후 로그인 오류가 늘었는지 확인해줘.
```

에이전트의 흐름은 다음처럼 될 수 있다.

```text
1. 목표 이해
최근 배포 이후 로그인 오류 증가 여부를 확인한다.

2. 계획 수립
- 최근 배포 시간 확인
- 배포 전후 로그인 API 에러율 비교
- 주요 에러 코드 확인
- 관련 로그 샘플 확인

3. 도구 실행
- 배포 이력 조회
- 모니터링 지표 조회
- 로그 검색

4. 결과 관찰
- 배포 이후 401 오류는 비슷함
- 500 오류가 증가함
- 특정 DB timeout 로그 증가

5. 추가 확인
- DB 연결 수 지표 조회
- 최근 DB 설정 변경 여부 확인

6. 최종 정리
- 배포 이후 로그인 API 500 오류 증가 확인
- DB timeout 증가와 시간대가 일치
- 원인 후보와 추가 확인 항목 제안
```

이 흐름을 보면 에이전트는 단순히 한 번 답하는 구조가 아니다.

중간 결과를 보고 다음 행동을 결정한다.

이런 구조를 `계획 → 실행 → 관찰 → 다음 실행`의 반복으로 볼 수 있다.

```text
계획
→ 실행
→ 관찰
→ 계획 수정
→ 실행
→ 관찰
→ 최종 답변
```

> 관찰은 도구 실행 결과를 확인하는 단계다.  
> 에이전트는 관찰 결과를 바탕으로 다음에 어떤 도구를 사용할지 결정할 수 있다.

## 5. 에이전트와 자동화의 차이

에이전트와 기존 자동화는 비슷해 보인다.

예를 들어 매일 아침 장애 로그를 요약하는 기능이 있다고 해보자.

기존 자동화는 정해진 순서대로 동작한다.

```text
매일 09:00 실행
→ 로그 조회
→ 에러 수 집계
→ 보고서 생성
→ Slack 전송
```

이 방식은 명확하고 안정적이다.

하지만 흐름이 고정되어 있다.

반면 에이전트는 상황에 따라 다음 행동을 바꿀 수 있다.

```text
로그 조회
→ 특정 에러 증가 발견
→ 관련 배포 이력 확인
→ 관련 문서 검색
→ 원인 후보 정리
```

차이를 정리하면 다음과 같다.

| 구분     | 기존 자동화           | AI 에이전트           |
|--------|------------------|-------------------|
| 동작 방식  | 정해진 절차 수행        | 목표를 보고 필요한 절차 선택  |
| 유연성    | 낮음               | 높음                |
| 예측 가능성 | 높음               | 상대적으로 낮음          |
| 적합한 작업 | 반복적이고 규칙이 명확한 작업 | 상황 판단이 필요한 탐색형 작업 |
| 위험도    | 상대적으로 낮음         | 도구 권한에 따라 커질 수 있음 |

기존 자동화가 더 적합한 경우도 많다.

예를 들어 매일 정해진 쿼리를 실행해서 지표를 보내는 일은 굳이 에이전트가 필요 없다.

```text
정해진 입력
정해진 절차
정해진 출력
```

이런 작업은 일반 배치나 스케줄러가 더 안정적이다.

에이전트가 적합한 경우는 다음과 같다.

```text
- 확인해야 할 정보가 상황마다 다르다.
- 여러 도구를 조합해야 한다.
- 중간 결과에 따라 다음 행동이 바뀐다.
- 사람이 하던 탐색형 업무를 보조하고 싶다.
```

에이전트는 모든 자동화를 대체하는 것이 아니다.

정해진 자동화 위에, 상황 판단이 필요한 부분을 보조하는 구조로 보는 것이 좋다.

## 6. 에이전트가 사용할 수 있는 도구

에이전트는 도구를 사용할 때 강력해진다.

도구가 없으면 AI는 설명만 할 수 있다.

도구가 있으면 실제 시스템에서 정보를 가져오거나 작업을 실행할 수 있다.

개발 업무에서 유용한 도구는 다음과 같다.

### 6.1 문서 검색 도구

```text
역할:
사내 문서, API 문서, 운영 매뉴얼을 검색한다.

사용 예:
- 환불 정책 문서 찾기
- 방송 시작 API 문서 찾기
- Redis 장애 대응 매뉴얼 찾기
```

### 6.2 로그 검색 도구

```text
역할:
운영 로그나 에러 로그를 검색한다.

사용 예:
- 최근 1시간 timeout 로그 조회
- 특정 userId 관련 요청 추적
- 특정 에러 코드 발생량 확인
```

### 6.3 모니터링 조회 도구

```text
역할:
서버, DB, Redis, API 지표를 조회한다.

사용 예:
- CPU 사용률 확인
- API 에러율 확인
- 응답 시간 증가 여부 확인
- DB 커넥션 수 확인
```

### 6.4 GitHub 도구

```text
역할:
PR, commit, diff, issue를 조회하거나 댓글을 작성한다.

사용 예:
- PR 변경 내용 요약
- 테스트 누락 확인
- 리뷰 코멘트 초안 생성
```

### 6.5 Jira 도구

```text
역할:
이슈, 담당자, 상태, 스프린트 정보를 조회하거나 업데이트한다.

사용 예:
- 이번 주 진행 중 이슈 요약
- 특정 에픽의 진행률 정리
- 지연 이슈 목록 확인
```

### 6.6 DB 조회 도구

```text
역할:
허용된 범위 안에서 데이터를 조회한다.

사용 예:
- 특정 문의 상태 확인
- AI 요약 결과 조회
- 배치 작업 결과 확인
```

DB 조회 도구는 특히 조심해야 한다.

읽기 전용부터 시작하고, 개인정보와 권한을 강하게 제한해야 한다.

### 6.7 알림 도구

```text
역할:
Slack, 메일, 알림 시스템으로 결과를 전송한다.

사용 예:
- 장애 요약 전송
- PR 리뷰 결과 알림
- 일일 리포트 발송
```

도구가 많아질수록 에이전트가 할 수 있는 일도 많아진다.

하지만 동시에 위험도 커진다.

그래서 도구는 필요한 것부터 작게 열고, 권한을 엄격하게 제한해야 한다.

---

## 7. 에이전트에게 모든 권한을 주면 위험하다

에이전트가 강력해질수록 위험도 커진다.

도구를 통해 실제 시스템에 접근할 수 있기 때문이다.

예를 들어 에이전트가 다음 권한을 가진다고 해보자.

```text
- DB 조회
- DB 수정
- 고객에게 메일 발송
- 관리자 권한 변경
- 계정 정지
- 환불 처리
- 배포 실행
```

이런 권한을 아무 제한 없이 주면 매우 위험하다.

AI가 사용자의 의도를 잘못 이해하거나, 잘못된 정보를 바탕으로 작업을 실행할 수 있기 때문이다.

예를 들어 사용자가 이렇게 말했다고 하자.

```text
이 사용자 문제 있는 것 같은데 처리해줘.
```

에이전트가 이를 계정 정지로 해석하고 바로 실행하면 안 된다.

안전한 구조는 다음과 같다.

```text
AI가 제안
→ 사람이 확인
→ 사람이 승인
→ 시스템이 실행
```

에이전트 권한은 단계적으로 열어야 한다.

```text
1단계:
읽기 전용 도구만 허용

2단계:
초안 생성 허용

3단계:
사람 승인 후 실행

4단계:
낮은 위험 작업만 자동 실행

5단계:
고위험 작업은 항상 승인 필요
```

작업을 위험도별로 나누면 좋다.

| 위험도   | 예시                      | 권장 방식       |
|-------|-------------------------|-------------|
| 낮음    | 문서 검색, 로그 요약, PR 요약     | 자동 실행 가능    |
| 중간    | Jira 댓글 작성, 내부 메모 초안 저장 | 검토 후 실행 권장  |
| 높음    | 고객 메일 발송, 권한 변경, 계정 제재  | 사람 승인 필수    |
| 매우 높음 | 환불, 결제 취소, 배포 실행, DB 수정 | 강한 승인 절차 필요 |

에이전트는 “할 수 있는 일”보다 “해도 되는 일”을 먼저 정해야 한다.

---

## 8. 읽기 도구와 쓰기 도구를 구분한다

에이전트 도구는 크게 읽기 도구와 쓰기 도구로 나눌 수 있다.

읽기 도구는 정보를 조회한다.

```text
- 문서 검색
- 로그 조회
- 모니터링 조회
- GitHub PR 조회
- Jira 이슈 조회
```

쓰기 도구는 시스템 상태를 변경한다.

```text
- Jira 이슈 상태 변경
- GitHub PR 댓글 작성
- 메일 발송
- DB 데이터 수정
- 계정 정지
- 환불 처리
```

읽기 도구도 민감할 수 있지만, 쓰기 도구는 훨씬 더 조심해야 한다.

쓰기 도구는 실제 결과를 남긴다.

잘못 실행하면 되돌리기 어렵거나, 고객에게 직접 영향을 줄 수 있다.

따라서 에이전트 설계에서는 읽기와 쓰기를 분리해야 한다.

```text
읽기 도구:
기본적으로 허용하되 권한 필터 적용

쓰기 도구:
기본적으로 제한
사람 승인 후 실행
실행 로그 필수
```

예를 들어 Jira 이슈 요약은 읽기 작업이다.

```text
Jira 이슈 조회
→ 상태와 담당자 정리
→ 보고서 생성
```

하지만 Jira 이슈 상태 변경은 쓰기 작업이다.

```text
Jira 이슈 상태를 Done으로 변경
```

이 작업은 사용자의 명확한 지시와 권한 확인이 필요하다.

또 메일 발송도 쓰기 작업이다.

AI가 초안을 만들 수는 있지만, 바로 보내는 것은 위험할 수 있다.

```text
안전한 방식:
AI가 메일 초안 작성
→ 사용자가 검토
→ 사용자가 발송 승인
```

도구를 설계할 때는 항상 이 질문을 해야 한다.

```text
이 도구는 정보를 읽기만 하는가?
아니면 실제 상태를 변경하는가?
```

상태를 변경한다면 더 강한 통제가 필요하다.

---

## 9. 에이전트의 작업 범위를 제한해야 한다

좋은 에이전트는 무엇이든 다 할 수 있는 AI가 아니다.

정해진 업무 범위 안에서 안정적으로 동작하는 AI다.

예를 들어 “개발팀 PR 리뷰 보조 에이전트”라면 범위를 다음처럼 정할 수 있다.

```text
가능한 일:
- PR diff 조회
- 변경 파일 요약
- 테스트 누락 후보 찾기
- 보안 위험 후보 찾기
- 리뷰 코멘트 초안 작성

불가능한 일:
- PR 자동 merge
- 배포 실행
- 코드 직접 수정
- 승인 없이 리뷰 코멘트 등록
```

범위가 명확하면 안전하다.

반대로 범위가 너무 넓으면 위험하다.

```text
나쁜 목표:
개발 업무를 알아서 처리하는 에이전트

좋은 목표:
PR 변경 내용을 요약하고 리뷰어가 확인할 포인트를 정리하는 에이전트
```

장애 대응 에이전트도 마찬가지다.

```text
가능한 일:
- 로그 조회
- 모니터링 지표 조회
- 최근 배포 이력 확인
- 원인 후보 정리
- 체크리스트 제안

불가능한 일:
- 서버 재시작
- 배포 롤백
- DB 수정
- 고객 공지 발송
```

처음에는 읽기와 분석 중심으로 제한하는 것이 좋다.

에이전트가 안정적으로 동작하고, 팀이 신뢰할 수 있게 되면 조금씩 범위를 넓힐 수 있다.

## 10. 에이전트는 근거를 남겨야 한다

에이전트가 도구를 사용했다면, 최종 답변에는 근거가 있어야 한다.

예를 들어 에이전트가 장애 원인 후보를 정리했다고 하자.

나쁜 답변은 다음과 같다.

```text
Redis 문제가 원인으로 보입니다.
```

이 답변은 너무 단정적이고 근거가 부족하다.

좋은 답변은 다음과 같다.

```text
## 확인된 사실
- 14:05부터 방송방 입장 API의 timeout이 증가했습니다.
- 같은 시간대 Redis 응답 시간이 증가했습니다.
- 최근 30분 내 관련 API 배포는 확인되지 않았습니다.

## 원인 후보
Redis 응답 지연으로 인해 방송방 입장 API timeout이 증가했을 가능성이 있습니다.

## 근거
- 모니터링 지표: broadcast_join_api timeout 증가
- Redis latency 지표: 14:05 이후 증가
- 배포 이력: 해당 시간대 배포 없음

## 추가 확인 필요
- Redis slowlog
- 특정 key 패턴 집중 여부
- 네트워크 지연 여부
```

에이전트가 사용한 도구와 결과를 사용자가 확인할 수 있어야 한다.

```text
- 어떤 로그를 조회했는가?
- 어떤 문서를 참고했는가?
- 어떤 PR을 확인했는가?
- 어떤 지표를 봤는가?
- 무엇은 확인하지 못했는가?
```

근거가 없으면 에이전트 답변은 신뢰하기 어렵다.

특히 장애, 보안, 비용, 운영 조치와 관련된 에이전트는 근거를 반드시 남겨야 한다.

## 11. 에이전트의 메모리와 상태 관리

에이전트는 작업 중 상태를 관리해야 할 수 있다.

간단한 챗봇은 대화 이력만 있어도 된다.

하지만 에이전트는 중간 작업 결과를 기억해야 한다.

예를 들어 장애 분석 에이전트가 다음 작업을 한다고 해보자.

```text
1. 에러율 조회
2. 에러 코드별 분포 확인
3. 최근 배포 이력 확인
4. 관련 로그 검색
5. 원인 후보 정리
```

각 단계의 결과를 저장해야 다음 단계에서 사용할 수 있다.

```text
에러율:
14:05 이후 증가

주요 에러 코드:
TIMEOUT

최근 배포:
없음

로그 특징:
Redis timeout 증가
```

이 상태를 바탕으로 최종 답변을 만든다.

에이전트의 상태는 크게 두 종류로 나눌 수 있다.

```text
단기 상태:
현재 작업을 수행하는 동안 필요한 정보

장기 메모리:
사용자나 조직에 대해 오래 유지할 정보
```

대부분의 업무 에이전트는 처음에는 단기 상태만으로 충분하다.

장기 메모리는 신중하게 사용해야 한다.

사용자 선호나 프로젝트 설정을 기억하는 것은 도움이 될 수 있지만, 민감 정보나 오래된 정보를 잘못 기억하면 문제가 될 수 있다.

예를 들어 다음 정보는 장기 메모리로 조심해야 한다.

```text
- 특정 사용자의 개인정보
- 임시 장애 원인 추정
- 오래된 정책
- 한 번만 필요한 API Key
- 민감한 내부 의사결정
```

에이전트는 무엇을 기억할지보다, 무엇을 기억하지 말아야 할지도 중요하다.

## 12. 에이전트의 실패 상황

에이전트는 여러 도구를 사용하기 때문에 실패 지점이 많다.

```text
- 도구 호출 실패
- 권한 부족
- 검색 결과 없음
- API timeout
- 잘못된 도구 선택
- 중간 결과 해석 오류
- 사용자의 의도 오해
- 작업 범위 초과
```

예를 들어 사용자가 이렇게 요청했다.

```text
최근 결제 오류 원인 확인해줘.
```

에이전트가 로그 검색 도구를 호출했는데 실패할 수 있다.

```text
로그 검색 실패:
권한 없음
```

이때 에이전트는 추측해서 답하면 안 된다.

좋은 답변은 다음과 같다.

```text
결제 오류 원인을 확인하려고 했지만, 로그 검색 권한이 없어 관련 로그를 조회하지 못했습니다.

현재 확인 가능한 정보:
- 결제 API 모니터링 지표는 조회 가능
- 로그 상세는 확인 불가

추가 필요:
- 로그 조회 권한 부여
- 또는 담당자가 결제 오류 로그를 제공
```

도구 호출 실패는 사용자에게 명확히 알려야 한다.

또 에이전트가 할 수 없는 작업을 요청받을 수도 있다.

```text
이 고객 환불 처리해줘.
```

환불은 고위험 작업이다.

에이전트가 직접 처리하면 안 된다.

안전한 답변은 다음과 같다.

```text
환불 처리는 고객 결제 상태와 정책 확인이 필요한 고위험 작업이므로 자동으로 실행하지 않습니다.

대신 환불 검토에 필요한 확인 항목을 정리하겠습니다.

- 결제 승인 여부
- 하트 사용 여부
- 환불 가능 기간
- 중복 결제 여부
- 고객 문의 내용
```

에이전트는 실패하거나 제한될 때도 안전하게 동작해야 한다.

---

## 13. 에이전트와 사람 승인 구조

에이전트가 쓰기 작업을 수행할 때는 사람 승인 구조가 중요하다.

이를 Human-in-the-loop라고 부를 수 있다.

핵심은 AI가 바로 실행하지 않고, 사람이 중간에서 확인하는 것이다.

```text
AI 제안
→ 사람 검토
→ 승인
→ 실행
```

예를 들어 고객 답변 초안 기능을 보자.

```text
1. AI가 고객 답변 초안을 작성한다.
2. 상담원이 내용을 확인한다.
3. 상담원이 수정한다.
4. 상담원이 발송 버튼을 누른다.
```

이 구조에서는 AI가 직접 고객에게 답변하지 않는다.

상담원이 최종 책임을 가진다.

Jira 상태 변경도 마찬가지다.

```text
AI:
이 이슈는 완료된 것으로 보입니다. Done으로 변경할까요?

사용자:
응, 변경해.

시스템:
사용자 권한 확인
→ Jira 상태 변경
→ 변경 로그 기록
```

사람 승인 구조에서는 승인 화면이나 확인 메시지가 중요하다.

```text
에이전트가 실행하려는 작업:
- Jira 이슈 PAND-123 상태를 In Progress에서 Done으로 변경
- 변경 사유: 연결된 PR이 merge됨
- 실행 사용자: admin_100

승인하시겠습니까?
```

사용자는 무엇이 실행되는지 명확히 알아야 한다.

고위험 작업일수록 승인 절차가 더 강해야 한다.

```text
낮은 위험:
확인 버튼 한 번

중간 위험:
작업 내용 검토 후 승인

높은 위험:
권한 확인 + 사유 입력 + 감사 로그

매우 높은 위험:
2인 승인 또는 별도 관리 시스템 처리
```

에이전트 자동화는 사람을 완전히 빼는 것이 목표가 아니다.

위험한 작업에서는 사람이 최종 판단하도록 만드는 것이 더 현실적이고 안전하다.

> Human-in-the-loop는 AI 결과나 실행 작업을 사람이 검토하고 승인하는 구조다.  
> 고객 영향이나 금전적 영향이 있는 작업에서는 특히 중요하다.

---

## 14. 에이전트 로그와 감사 추적

에이전트가 도구를 사용하면 반드시 로그를 남겨야 한다.

특히 쓰기 작업을 수행할 수 있다면 감사 추적이 필요하다.

감사 추적은 나중에 “누가, 언제, 무엇을, 왜 실행했는지” 확인할 수 있게 하는 것이다.

에이전트 로그에는 다음 정보가 필요하다.

```text
- 요청 ID
- 사용자 ID
- 사용자 권한
- 사용자 요청
- 에이전트가 세운 계획
- 호출한 도구
- 도구 입력값
- 도구 실행 결과
- 최종 답변
- 실행된 쓰기 작업
- 승인자
- 실행 시간
```

예를 들어 Jira 상태 변경을 에이전트가 도왔다면 다음 로그가 필요하다.

```json
{
  "requestId": "agent_req_123",
  "userId": "admin_100",
  "action": "jira_status_update",
  "target": "PAND-123",
  "beforeStatus": "In Progress",
  "afterStatus": "Done",
  "approvedBy": "admin_100",
  "reason": "연결된 PR merge 확인",
  "createdAt": "2026-05-04T10:30:00+09:00"
}
```

로그는 문제 발생 시 매우 중요하다.

```text
고객 답변이 잘못 발송됨
→ AI가 어떤 초안을 만들었는지 확인
→ 누가 승인했는지 확인
→ 어떤 문서를 근거로 했는지 확인

Jira 상태가 잘못 변경됨
→ 어떤 요청으로 변경됐는지 확인
→ 승인자가 누구인지 확인
→ 변경 전 상태 확인
```

에이전트는 여러 도구를 조합하기 때문에 일반 기능보다 추적이 더 중요하다.

로그가 없으면 에이전트의 판단 과정을 나중에 확인하기 어렵다.

---

## 15. 에이전트를 처음 도입할 때 추천하는 범위

에이전트는 강력하지만 처음부터 크게 만들면 위험하다.

처음에는 읽기 전용, 내부 보조 업무부터 시작하는 것이 좋다.

추천하는 1차 범위는 다음과 같다.

```text
1. 문서 검색 에이전트
- 사내 문서를 검색하고 출처와 함께 답변
- 쓰기 작업 없음

2. PR 리뷰 보조 에이전트
- PR diff 조회
- 위험 후보 정리
- 댓글은 초안만 생성
- 자동 등록 없음

3. 장애 분석 보조 에이전트
- 로그와 지표 조회
- 원인 후보 정리
- 서버 재시작이나 롤백 없음

4. Jira 리포트 에이전트
- 이슈 조회
- 진행 상황 요약
- 상태 변경 없음
```

처음부터 피해야 할 범위는 다음과 같다.

```text
- 자동 환불 처리
- 계정 정지 자동 실행
- 고객 메일 자동 발송
- DB 수정
- 배포 자동 실행
- 관리자 권한 자동 부여
```

에이전트 도입은 다음 순서가 안전하다.

```text
1단계:
읽기 전용 도구 연결

2단계:
초안 생성

3단계:
사람 승인 후 쓰기 작업

4단계:
낮은 위험 작업만 제한적 자동 실행

5단계:
감사 로그와 모니터링 강화
```

처음에는 “AI가 대신 처리하는 시스템”보다 “AI가 확인과 정리를 도와주는 시스템”으로 시작하는 것이 좋다.

## 16. 에이전트 설계 체크리스트

에이전트를 설계할 때는 다음 체크리스트를 사용할 수 있다.

### 16.1 목표와 범위

```text
- 에이전트의 목적이 명확한가?
- 어떤 업무를 도와주는가?
- 하지 말아야 할 작업이 정의되어 있는가?
- 사용자 역할별로 가능한 작업이 다른가?
```

### 16.2 도구

```text
- 어떤 도구를 사용할 수 있는가?
- 각 도구는 읽기 도구인가, 쓰기 도구인가?
- 도구별 권한이 제한되어 있는가?
- 도구 입력값 검증이 있는가?
```

### 16.3 권한과 승인

```text
- 사용자의 권한을 확인하는가?
- 쓰기 작업은 사람 승인을 받는가?
- 고위험 작업은 별도 승인 절차가 있는가?
- 승인 전 실행 내용을 명확히 보여주는가?
```

### 16.4 안전장치

```text
- 작업 범위 초과 요청을 거절하는가?
- 도구 실패 시 추측하지 않는가?
- 민감 정보가 로그에 남지 않는가?
- 잘못된 실행을 되돌릴 수 있는가?
```

### 16.5 근거와 로그

```text
- 에이전트가 어떤 도구를 사용했는지 기록하는가?
- 최종 답변에 근거를 포함하는가?
- 실행 로그와 승인 로그를 남기는가?
- 문제 발생 시 추적 가능한가?
```

이 체크리스트를 보면 알 수 있듯이 에이전트 설계는 단순히 AI 모델을 고르는 일이 아니다.

도구, 권한, 승인, 로그, 실패 처리까지 함께 설계해야 한다.

## 17. 정리

AI 에이전트는 단순히 질문에 답하는 챗봇보다 한 단계 더 나아간 구조다.

에이전트는 목표를 이해하고, 필요한 도구를 선택하고, 여러 단계를 실행하며, 결과를 정리한다.

개발 업무에서는 PR 리뷰 보조, 장애 분석, Jira 리포트, 사내 문서 검색, 운영 매뉴얼 확인 같은 영역에서 활용할 수 있다.

하지만 에이전트는 강력한 만큼 위험도 크다.

도구를 통해 실제 시스템에 접근할 수 있고, 쓰기 작업을 수행할 수도 있기 때문이다.

그래서 에이전트 설계에서는 다음이 중요하다.

```text
- 작업 범위를 명확히 제한한다.
- 읽기 도구와 쓰기 도구를 구분한다.
- 권한을 서버에서 검증한다.
- 고위험 작업은 사람 승인을 받는다.
- 실행 근거와 로그를 남긴다.
- 실패하거나 모를 때 추측하지 않는다.
```

처음에는 읽기 전용 에이전트부터 시작하는 것이 좋다.

문서를 찾고, 로그를 요약하고, PR을 분석하고, 이슈 현황을 정리하는 정도부터 시작하면 안전하게 효과를 확인할 수 있다.

이 장에서 기억해야 할 핵심은 하나다.

> AI 에이전트는 “AI가 알아서 다 해주는 자동화”가 아니다.  
> 목표, 도구, 권한, 승인, 로그를 갖춘 안전한 업무 보조 구조다.
